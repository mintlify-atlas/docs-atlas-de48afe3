The Hostel Management System uses the browser's localStorage API to persist resident and room data. This eliminates the need for a backend database while maintaining data across browser sessions.

## Storage overview

The system stores data in two localStorage keys:

<CardGroup cols={2}>
  <Card title="rooms" icon="door-open">
    Array of room objects with occupancy status. Contains 10 rooms (101-110) by default.
  </Card>
  
  <Card title="residents" icon="users">
    Array of resident records with personal information and room assignments.
  </Card>
</CardGroup>

## Implementation

The storage implementation is contained in the `hostelService` class at `src/services/hostelService.ts`.

### Loading data

Data is loaded when the service is instantiated:

```typescript
constructor() {
  this.loadData();
}

loadData(): void {
  const storedRooms = localStorage.getItem("rooms");
  const storedResidents = localStorage.getItem("residents");
  if (storedRooms) {
    this.rooms = JSON.parse(storedRooms);
  } else {
    this.rooms = roomsAvailability;
  }
  if (storedResidents) {
    this.resident = JSON.parse(storedResidents);
  } else {
    this.resident = [];
  }
}
```

See `src/services/hostelService.ts:9` for the constructor and `src/services/hostelService.ts:14` for the `loadData()` method.

<Info>
On first load, if no stored data exists, the system initializes with default room data from `src/data/roomsData.ts` and an empty residents array.
</Info>

### Saving data

Data is automatically saved after every state modification:

```typescript
saveData() {
  localStorage.setItem("rooms", JSON.stringify(this.rooms));
  localStorage.setItem("residents", JSON.stringify(this.resident));
}
```

See `src/services/hostelService.ts:38` for the implementation.

<Warning>
The `saveData()` method is called at the end of every operation that modifies state (add, update, remove). Forgetting to call this method would result in data loss on page refresh.
</Warning>

## Storage capacity

The code includes a comment about localStorage capacity:

```typescript
//web storage->browser gives storage of 5mb in json formate. This is also a web API
```

See `src/services/hostelService.ts:12`.

<Note>
Most browsers provide 5-10MB of localStorage space per origin. For a hostel management system with hundreds of residents, this is more than sufficient. Each resident record is approximately 150-200 bytes.
</Note>

### Capacity estimation

For typical usage:

| Item | Size per record | Max records (5MB) |

| Resident | ~200 bytes | ~25,000 residents |
| Room | ~50 bytes | ~100,000 rooms |

## Data initialization

The system uses a fallback pattern for initial data:

```typescript
if (storedRooms) {
  this.rooms = JSON.parse(storedRooms);
} else {
  this.rooms = roomsAvailability; // Default from roomsData.ts
}
if (storedResidents) {
  this.resident = JSON.parse(storedResidents);
} else {
  this.resident = []; // Empty array
}
```

This ensures:
- **First load**: Rooms are initialized from `src/data/roomsData.ts`, residents start empty
- **Subsequent loads**: Data is loaded from localStorage
- **After clearing storage**: System resets to initial state

## Clearing stored data

To reset the application to its initial state:

<CodeGroup>
```javascript Browser console
// Clear all hostel data
localStorage.removeItem('rooms');
localStorage.removeItem('residents');

// Or clear all localStorage
localStorage.clear();

// Then refresh the page
location.reload();
```

```javascript Programmatically
// Add a reset method to hostelService class
resetData() {
  localStorage.removeItem('rooms');
  localStorage.removeItem('residents');
  this.loadData(); // Reload with defaults
}
```
</CodeGroup>

## Limitations and considerations

<AccordionGroup>
  <Accordion title="No server synchronization">
    Data is stored only in the browser. It cannot be accessed from other devices or browsers, and is lost if browser data is cleared.
  </Accordion>
  
  <Accordion title="No backup mechanism">
    There's no automatic backup. Users should export data if they need to preserve it outside the browser.
  </Accordion>
  
  <Accordion title="Single-user only">
    localStorage is per-browser, so the system doesn't support multiple concurrent users or collaboration.
  </Accordion>
  
  <Accordion title="No data migration">
    Changing data structures requires manual migration or clearing existing data. There's no built-in versioning system.
  </Accordion>
  
  <Accordion title="Browser limitations">
    Private/incognito mode, browser extensions, or security settings may restrict localStorage access.
  </Accordion>
</AccordionGroup>

<Warning>
Users browsing in private/incognito mode will lose all data when closing the browser, as localStorage is cleared on session end in these modes.
</Warning>

## Benefits of localStorage

Despite limitations, localStorage offers several advantages for this application:

<CardGroup cols={2}>
  <Card title="Zero backend" icon="server">
    No server setup, hosting, or database management required. The application is completely self-contained.
  </Card>
  
  <Card title="Instant persistence" icon="bolt">
    Data is saved synchronously with no network latency. All operations complete immediately.
  </Card>
  
  <Card title="Offline-first" icon="wifi">
    Works completely offline. No internet connection required after initial page load.
  </Card>
  
  <Card title="Simple API" icon="code">
    Just `setItem()` and `getItem()`. No query languages, ORMs, or connection management.
  </Card>
</CardGroup>

## Migration to backend storage

If you need to migrate to a backend database later:

<Steps>
  <Step title="Extract storage interface">
    Create an interface for storage operations:
    ```typescript
    interface Storage {
      loadRooms(): Promise<Rooms[]>;
      loadResidents(): Promise<Resident[]>;
      saveRooms(rooms: Rooms[]): Promise<void>;
      saveResidents(residents: Resident[]): Promise<void>;
    }
    ```
  </Step>
  
  <Step title="Implement localStorage adapter">
    Wrap current implementation in the interface:
    ```typescript
    class LocalStorageAdapter implements Storage {
      async loadRooms(): Promise<Rooms[]> {
        const stored = localStorage.getItem('rooms');
        return stored ? JSON.parse(stored) : roomsAvailability;
      }
      // ... other methods
    }
    ```
  </Step>
  
  <Step title="Create backend adapter">
    Implement the same interface with API calls:
    ```typescript
    class BackendStorageAdapter implements Storage {
      async loadRooms(): Promise<Rooms[]> {
        const response = await fetch('/api/rooms');
        return response.json();
      }
      // ... other methods
    }
    ```
  </Step>
  
  <Step title="Update service layer">
    Inject the storage adapter and update methods to use async/await. The business logic remains unchanged.
  </Step>
</Steps>

<Info>
The current synchronous storage pattern would need to become asynchronous when moving to a backend API. This would require updating all service methods to return Promises.
</Info>
